# 2026-05-25 — feat: атомарные игровые действия

## Цель

Реализовать фичу [atomic-actions](../features/done/atomic-actions.md): сделать команды
`MOVE_TO`, `MINE`, `DROP`, `CHARGE` атомарными — одна команда = одна единица работы.
Макроповедение (заряжаться до полного, копать пока влезает) игрок собирает сам через
`LOOP` / `REPEAT` / `IF`.

## Что сделано

- **MovementSystem** — `MOVE_TO` теперь делает ровно один шаг по A*-пути и зачищает
  остаток `movement.path`. Энергия тратится один раз за команду, даже при `speed >= 10`.
- **MiningSystem** — `MINE` и `DROP` дают по +1 руды за вызов и сразу возвращают программу
  в `running`. По пути починен баг: при пустом инвентаре `DROP` уводил `ore` в −1
  (был покрыт регрессионным тестом).
- **EnergySystem** — активная `CHARGE` даёт +1 и завершается. Пассивная зарядка на станции
  (без `waitingFor=charge`) сохранена как свойство станции, не команды.
- **Интеграционный тест** `src/game/simulation/atomic-actions.integration.test.ts` —
  доказывает, что `LOOP { MINE }` добывает руду порционно: ровно по одной за 21 тик
  (push LOOP-кадра + выдача MINE + 19 тиков накопления).
- **DECISIONS.md** — добавлена запись «действия дронов атомарны».
- **Фича** перенесена `planned/atomic-actions.md → done/atomic-actions.md`, индекс обновлён.

## Известное следствие — миссии 1–4 ломаются по смыслу

Программы и тексты подсказок встроенных миссий (`src/game/missions/mission{1..4}.ts`)
построены на старой семантике «команда доводит работу до конца» (`LOOP { MOVE_TO →
MINE → MOVE_TO → DROP }`). После этой фичи такие программы продолжают запускаться,
но ведут себя неожиданно. Миграция миссий — отдельная фича `missions-atomic-migration`
(пока не заведена). Готовая заготовка промпта — в конце
[docs/features/done/atomic-actions.md](../features/done/atomic-actions.md).

## Процесс

Использован subagent-driven подход для трёх первых задач (MovementSystem, MiningSystem,
EnergySystem) с двухстадийным ревью (spec + code quality) на каждую. Задачи 4–6
(интеграционный тест, документация, верификация) — inline.

Subagent в Task 2 пропустил защиту от пустого инвентаря при `DROP` — поймано на spec
review, починено отдельным `fix:` коммитом. Subagent в Task 3 заметил ошибку в самом
плане (ожидание `energy=51` после 10 тиков было несовместимо с сохранением пассивной
зарядки) и корректно его исправил — тест переписан, чтобы проверять и атомарность
команды, и пассивную зарядку. Один code-quality nit (мутная mixed-проверка в
`EnergySystem.update`) починен отдельным `refactor:` коммитом.
