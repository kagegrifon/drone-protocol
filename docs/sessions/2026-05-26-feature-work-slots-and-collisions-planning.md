# 2026-05-26 — feature: планирование «эксклюзивные слоты работы и физическая непроходимость»

## Цель

Спланировать новую фичу: добывать и заряжаться можно только по одному (логическая ёмкость), и роботы не могут наезжать друг на друга (физическая непроходимость). Архитектура должна быть готова к будущим многоклеточным шахтам с подходами сбоку. Реализация — в отдельной сессии.

## Что сделано

- Изучён текущий код: `CollisionSystem`, `MovementSystem`, `MiningSystem`, `EnergySystem`, `interpreter.ts`, реестр функций `FUNCTIONS`, компоненты `Deposit`/`ChargerStation`.
- Через брейнсторминг (`superpowers:brainstorming`) проведено 8 итераций уточнений:
  1. Что значит «по одному»: и физическое, и логическое правило, с возможностью будущей «вертикальной загрузки».
  2. Геометрия: рассматривался переход на «работа с соседней клетки», в итоге выбран **гибрид** — дрон по-прежнему встаёт на клетку сущности, но в данных закладываются `workSlots` для будущего мульти-слота.
  3. Связность подходов (4 vs 8) — отложено, так как геометрия не меняется.
  4. Семантика `MOVE_TO` — не меняется, новый только `FreeSlots(target)` в IF.
  5. Закрытие физической дыры в `MovementSystem` (два дрона могли запланировать путь в одну клетку и оба туда шагнуть) — **включено в скоуп**.
  6. Открытие данных программам — функция `FreeSlots(target)` (числовая, единый интерфейс для текущей геометрии и будущей мульти-слотовой).
  7. Производное vs явное состояние занятости слота — выбрано **явное** (производное лишает программ возможности дешево опрашивать `FreeSlots` без обхода всех дронов).
  8. Lifecycle: занятость = физическое присутствие на клетке слота, а не активная фаза работы (иначе между двумя циклами `MINE` слот «мигал» бы как свободный, путая программы других дронов).
- Дизайн презентован по 8 разделам (обзор, модель данных, lifecycle, фикс гонки `MovementSystem`, `FreeSlots`, валидация спавна, тесты, файлы) с подтверждением после каждого.
- Написана спецификация [docs/features/planned/work-slots-and-collisions.md](../features/planned/work-slots-and-collisions.md).
- Обновлён [docs/features/index.md](../features/index.md): добавлена запись про `work-slots-and-collisions.md`.
- Self-review спеки: проверены placeholders, внутренняя согласованность, скоуп и двусмысленности. Найдена одна минорная (что считается «базой» при навешивании `WorkSlots`) — пометки уже корректны, дополнительных правок не требуется.

## Ключевые решения

- **Слот занят, пока дрон физически на его клетке.** Не привязано к `program.waitingFor` — иначе слот мигал бы как свободный между атомарными циклами `MINE`/`CHARGE`/`DROP`, и другие дроны принимали бы ложные решения по `FreeSlots()`.
- **Lifecycle через события, не через сканирование.** `MovementSystem` эмитит `drone:moved`, подписчик в модуле `workSlotsIndex` обновляет `occupiedBy` двумя O(1)-поисками по индексу `Map<"x,y", SlotRef>`. Никаких «дрон при движении пробегает все свои слоты», никаких «слоты сканируют дронов».
- **Публичный API entity-centric:** только `slotsOf(entity)` и `freeSlotsCount(entity)`. Никакого `getSlotAt(x, y)` или `releaseSlotByDrone(droneId)` наружу.
- **Фикс гонки в `MovementSystem` через локальный `stepped`-set.** В начале тика инициализируется текущими позициями, при шаге `release` старой клетки + `claim` новой. При попытке шагнуть в занятую клетку — отмена шага + `drone:blocked` (та же семантика, что в planned/congestion-metric).
- **Спавн на слоте запрещён.** Громкое падение при инициализации мира, без авто-сдвига.
- **`FreeSlots(target)` как числовая функция.** Стабильный API: 0/1 сейчас, 0..N в будущем. Программы, написанные сейчас, будут работать после введения мульти-слота.

## Что НЕ вошло в эту сессию

- Реализация фичи — пойдёт в отдельной сессии.
- Депозит/зарядка/база остаются проходимыми. Слоты сбоку и 4/8-связность подходов — будущая фича.
- Резервирование пути «дрон занял слот, ещё не доехав» — будущая фича.
- Dev-asserts на рассинхрон состояния слотов и `waitingFor` — отдельная фича.
- Починка существующих миссий, если они начнут падать из-за валидации спавна — отдельная сессия.
