# Сессия: Исправление e2e-тестов после рефакторинга блочной логики

**Дата:** 2026-06-18 19:36  
**Веточка:** `refactor/exclude-block-logic`  
**Статус:** ✅ Завершено

## Цель

Исправить падающие e2e-тесты после рефакторинга, который удалил блочную логику дронов (остался только Code mode). Было 5 упавших e2e-тестов, нужно сделать их зелёными.

## Корневые причины (Phase 1 — Systematic Debugging)

### Проблема 1: `code-mode.spec.ts:55` — несуществующий toggle

- **Симптом:** Тест ищет `[data-testid="code-mode-toggle-code"]`, которого больше нет.
- **Причина:** `enableCodeMode()` пытался переключить режим через UI модалки Audio Settings. После рефакторинга Code Mode — единственный и всегда активен.
- **Диагностика:** Прочитал текущий UI (`AudioSettingsModal.tsx`, `SimControls.tsx`) — там нет переключателя режимов.

### Проблема 2: `move-to-picker.spec.ts:16` — удалённый блочный компонент

- **Симптом:** Тест ищет `[data-testid="move-to-toggle"]` из `InstructionBlock.tsx`.
- **Причина:** Все блочные компоненты удалены в рефакторинге (`InstructionBlock.tsx`, `DropSlot.tsx`, ConditionEditor.tsx и др.).
- **Диагностика:** Компонент `ProgramEditor` больше не принимает проп `entities` и всегда показывает `CodeEditor`.

### Проблема 3: `drone-controls.spec.ts` (3 теста) — flaky load times

- **Симптом:** При параллельном запуске тесты застревают на LoadingScreen, падают с `timeout: 15000ms exceeded`.
- **Причина:** Phaser GPU-инициализация при параллельной нагрузке 6–8 тестов требует больше времени; таймауты слишком короткие.
- **Диагностика:** Тесты проходят изолированно, падают при совместном запуске → проблема ресурсов, не регрессия.

## Решения (Phase 4 — Implementation)

### Fix 1: Упростить `code-mode.spec.ts`

Убрал функцию `enableCodeMode()` и её вызов — тест сразу переходит к миссии без попытки переключить режим.

```diff
- async function enableCodeMode(page) { ... }
- await enableCodeMode(page);
  await startMission(page, 0);
```

### Fix 2: Удалить `move-to-picker.spec.ts`

Тест покрывал блочный UI, который больше не существует. Файл удалён полностью.

### Fix 3: Увеличить таймауты e2e-тестов

Увеличил таймауты ожидания загрузки игры:
- Canvas/Play button: 15s → **30s**
- Drone list: 10s → **20s**

Это позволяет Phaser загружаться при параллельной нагрузке без спешки.

### Fix 4: Ограничить параллельные workers

В `playwright.config.ts` добавил адаптивное количество workers:

```typescript
workers: process.env.CI ? undefined : 3,
```

- **Локально:** 3 worker'а (Phaser тяжёлый, GPU конкурирует)
- **GitHub Actions:** `undefined` (Playwright сам выбирает по CPU, обычно половина)

Это убирает перегрузку при параллельном тестировании.

## Ключевые выводы

1. **Flaky-тесты не всегда регрессия** — перед фиксом нужно изолировать тест и проверить, проходит ли он отдельно. Если проходит — проблема инфраструктуры (timeouts, resources), не кода.

2. **Phaser в Playwright требует ресурсов** — инициализация GPU-контекста в headless браузере конкурирует между тестами. Ограничение workers и increase timeouts — правильный подход.

3. **Систематический debugging спасает время** — вместо наугад менять таймауты, запустил диагностику (изолировано vs параллельно) и выяснил корневую причину за минуты.
