# Сессия: дизайн системы дебаггинга кода дронов

**Дата:** 2026-06-20
**Тип:** feature (дизайн/brainstorming, без реализации)
**Ветка:** feat/camera-focus (реализация пойдёт в отдельных ветках)

## Цель

Разработать систему дебаггинга кода дронов: показать пользователю, где находится
исполнение кода (подсветка текущей выполняемой команды), чтобы было понятнее, как
работает написанный код.

## Результаты

Проведён brainstorming, изучена архитектура исполнения кода. Ключевая находка:
код игрока компилируется как `AsyncFunction` из строки в worker'е — движок **не
знает номер исполняемой строки** (`currentInstruction` = `"-TODO-"`,
`CallFrame.instructionIndex` всегда `0`). Исполнение «видимо» только в точках
`await drone.*()`, где worker шлёт `intent` и зависает.

### Решения

- **Scope:** только подсветка текущей строки (без step/breakpoints/панели сенсоров).
- **Механизм определения строки:** инструментация кода через AST (`acorn`) —
  вставка маркера `__line(N)` перед каждым `await drone.*`. Выбран как надёжный
  (vs `Error().stack` — хрупкий из-за разных форматов стека).
- **UX подсветки:** только для активной программы выбранного дрона; привязка к
  вкладке DRONE (что там открыто, то и подсвечиваем).
- **Назначенная программа:** на вкладке DRONE её код должен быть виден и
  редактируем — симметрично персональной.
- **Разделение на 2 этапа** (по просьбе пользователя):
  1. UI: редактор кода назначенной программы на вкладке DRONE.
  2. Подсветка текущей строки (инструментация → проброс line → Monaco decoration).
- **Step-режим:** не реализуем сейчас. В коде инструментатора оставить комментарий
  о будущем расширении (инструментация каждого statement + `await __step(N)`), но
  БЕЗ параметризации сейчас.
