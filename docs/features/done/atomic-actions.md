# Атомарные игровые действия

**Статус:** done

## Зачем

Сейчас одно действие в программе («MOVE_TO», «MINE», «DROP», «CHARGE») доводится симуляцией «до конца»: `MOVE_TO` идёт по всему A*-пути до целевой сущности, `MINE` добывает руду пока депозит и инвентарь позволяют, `DROP` выгружает весь инвентарь, `CHARGE` заряжает до максимума. Это удобно для простых программ, но лишает игрока контроля: между «зарядкой первой единицы» и «зарядкой последней» нельзя вставить условие, проверку или другой блок.

Делаем все четыре действия атомарными — одна команда = одна единица работы. Макроповедение («заряжаться до полного», «копать пока влезает») игрок строит сам через `LOOP`, `REPEAT` и `IF`. Это раскрывает возможности существующих управляющих блоков и упрощает мысленную модель: «команда симуляции = один тик действия дрона».

## Поведение

Длительности и стоимости энергии — без изменений (см. `src/game/simulation/constants.ts`).

**`MOVE_TO target`** — один шаг по A*-пути в сторону целевой сущности. Тратится `energy.drainPerMove`. Как только дрон сдвинулся на одну клетку (или путь невалиден / пуст), команда завершается и программа переходит к следующей инструкции.

**`MINE`** — одна единица руды. После накопления `BASE_MINE_DURATION_PER_ORE` (2.0 c): −1 у депозита, +1 в инвентарь, тратится `drainPerMine`, эмитится событие `ore:mined`, команда завершается. Если дрон не на депозите, депозит пуст или инвентарь полон — команда завершается сразу, без расхода времени и энергии.

**`DROP`** — одна единица руды. После накопления `BASE_DROP_DURATION_PER_ORE` (0.5 c): −1 у дрона, +1 на базу, эмитится `ore:dropped` (`amount: 1`), команда завершается. Если дрон не на базе или инвентарь пуст — команда завершается сразу.

**`CHARGE`** — одна единица энергии. После накопления `BASE_CHARGE_DURATION_PER_UNIT` (0.5 c): +1 к `energy.current`, команда завершается. Если дрон не на станции или энергия уже полная — команда завершается сразу.

Во всех случаях «команда завершается» означает: интерпретатор переходит к следующей инструкции текущего фрейма. Если инструкций больше нет и call-стек пуст — состояние программы становится `idle`. Блок `WAIT { seconds }` — отдельная сущность для пауз между шагами, не затрагивается этой фичей.

## Критерии готовности

- Unit-тесты `MovementSystem`, `MiningSystem`, `EnergySystem` обновлены под новую семантику и проходят:
  - `MINE` даёт ровно +1 руду за вызов; повторный вызов добывает следующую.
  - `MINE` на полном инвентаре / пустом депозите / не на депозите — завершается сразу, без задержки и энергозатрат.
  - `DROP` даёт ровно +1 руду на базе за вызов; пустой инвентарь / не на базе — завершается сразу.
  - `CHARGE` даёт ровно +1 энергии за вызов; полная энергия / не на станции — завершается сразу.
  - `MOVE_TO` после одного достигнутого узла снимает `waitingFor` и не «доедает» оставшийся путь.
- Стоимости энергии и длительности тиков совпадают с константами в `src/game/simulation/constants.ts` — не меняются.
- Добавлен e2e-тест: программа `LOOP { MINE }` на депозите с N руды добывает по одной единице за `BASE_MINE_DURATION_PER_ORE` секунд (видна порционность, а не одномоментная выгрузка).
- Существующий e2e против цикла `MOVE_TO → MINE → MOVE_TO → DROP` ожидаемо ломается — это известно и фиксируется в отдельной фиче по миграции миссий (см. ниже).
- Запись в `DECISIONS.md`: «действия дронов атомарны — одна команда = одна единица работы».

## Технические заметки

- Изменения локальны: правки в `src/game/simulation/systems/MovementSystem.ts`, `MiningSystem.ts`, `EnergySystem.ts`. AST (`src/game/programs/types.ts`) и интерпретатор (`src/game/programs/interpreter.ts`) — без изменений.
- В `MovementSystem`: убрать цикл `while (movement.progress >= 1 …)`, оставить однократное продвижение по `path` за тик.
- В `MiningSystem.processMining` и `processDrop`: убрать циклы `while`, выходить из `waiting` сразу после одной обработанной единицы.
- В `EnergySystem`: команда `CHARGE` ждёт ровно одну порцию, +1 энергии, выход. **Пассивную зарядку** (дрон стоит на станции без активной `CHARGE` — текущее поведение в `EnergySystem.ts:25-31`) сознательно оставляем как есть: это свойство станции, не команды. Изменение пассивной зарядки — отдельная фича.
- Edge-cases (не на депозите/базе/станции, полный инвентарь, пустой депозит, полная энергия) — завершают команду без расхода времени; это и сейчас так работает.
- Сохранённые user-программы из прогресса миссий продолжат загружаться (AST не меняется), но проигрываться будут иначе. Игрокам придётся пересобрать макроповедение через `LOOP` / `IF`.

### Известное следствие: миссии 1–4 ломаются

Все встроенные миссии (`src/game/missions/mission{1..4}.ts`) и тексты подсказок построены на семантике «команда доводит работу до конца» (`LOOP { MOVE_TO(mine) → MINE → MOVE_TO(base) → DROP }`). После этой фичи такие программы продолжат запускаться, но будут вести себя неожиданно. Миграция миссий — отдельная фича, в скоуп этой не входит.

### Заготовка промпта для следующей фичи (миграция миссий)
