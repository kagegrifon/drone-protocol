# Continuous drone movement — дизайн

**Дата:** 2026-06-23
**Ветка:** fix/jerky-drone-movement
**Статус:** одобрен, готов к плану

## Проблема

Движение дрона дёрганое. История правок и их дефекты:

- **До `6078efc`** — одношаговая модель: `moveTo` делает 1 клетку, сбрасывает
  `path=[]`, возвращает управление (`state=running`). Программа в цикле снова
  вызывает `moveTo` → новый шаг. Даёт точку пересчёта поведения между клетками,
  **но** между шагом и приходом нового `intent` от воркера (~15ms async) дрон
  стоит 1 тик (~100ms) → видимая пауза/дёрганость на каждой клетке.

- **`6078efc`** — попытка look-ahead: убрал `path=[]`, заставил `MovementSystem`
  идти по всему сохранённому пути; `onDroneMoved` шлёт resume только при пустом
  path. Убрал паузу, **но сломал модель**: воркер не опрашивается по пути, дрон
  идёт весь маршрут слепо — игрок не может изменить поведение между клетками
  (поехать заряжаться, сменить шахту). Плюс визуальный «рывок назад».

## Как это решают в играх

Стандарт индустрии (RTS, Factorio) — **path following + replanning**: юнит
хранит путь и движется непрерывно; параллельно по событию/таймеру путь
пере-валидируется и корректируется на лету, **не останавливая движение**.
Особенность Drone Loop: «мозг» дрона — JS в воркере с async-latency ~15ms,
поэтому наивный «шаг → спросить воркер → ждать» даёт паузу, которой в нативных
играх нет.

## Целевая модель

**Дрон движется непрерывно, пока не получит сигнал остановиться.**

Сигнал «стоп» (плавная остановка на клетке) приходит когда:
- следующее действие ≠ `moveTo` (mine / charge / drop), **или**
- `moveTo` с **изменившейся целью** (другая `target`), **или**
- подтверждение курса не пришло к концу шага (cold start / лаг).

Если следующее действие — `moveTo` к **той же цели**, путь продлевается и дрон
**не тормозит** (вычисление быстрее анимации, успевает за время шага).

Критерий «та же траектория» — **по целевой точке** (`targetX/targetY`).
Механика хранения пути — **look-ahead на 1 шаг**: `path` содержит максимум
следующую 1 клетку; после каждого шага driver досчитывает следующую.

## Отправная точка (состояние рабочего дерева)

В рабочем дереве есть незакоммиченные правки, частично откатывающие `6078efc`
к одношаговой модели (`MovementSystem` снова делает `path=[]`; `CodeBehaviorDriver`
шлёт ранний resume всегда). План должен исходить из коммита `6078efc` как базы и
довести до целевой continuous-модели; незакоммиченный промежуточный откат можно
отбросить или переиспользовать как удобнее.

## Архитектура

### 1. MovementComponent
Без структурных изменений. `path: Position[]` снова может содержать следующий
шаг (продлеваемый буфер). `targetX/targetY` — текущая цель moveTo, по ней
сравнивается «та же траектория».

### 2. MovementSystem
Делает один шаг по `path[0]`. После шага:
- `path` сдвигается (`shift`).
- Если после shift `path` **не пуст** (driver успел дописать следующий шаг за
  время текущего) → `progress=0`, `state` остаётся `move`, движение
  продолжается **без паузы**.
- Если `path` **пуст** → известный путь исчерпан. `state=running` (если был
  `move`) → программа получит управление. Дрон стоит ровно на клетке
  (безопасная остановка).

`drone:moved` эмитится после каждого шага (как сейчас).

### 3. CodeBehaviorDriver
Обработка `intent: moveTo(point)`:
- Сравнить `point` с текущими `movement.targetX/targetY`.
- **Та же цель** и дрон в движении → распланировать **следующий 1 шаг** A* от
  текущей позиции к цели и **дописать** его в `path` (не сбрасывать `progress`).
  Дрон не тормозит.
- **Другая цель** или дрон стоит → построить новый путь (как сейчас через
  `planMoveToPoint`), дрон плавно поедет в новом направлении.

Ранний resume на `drone:moved` сохраняется: после шага resume шлётся сразу
(~15ms), воркер успевает вернуть `intent` до конца текущего шага → driver
дописывает следующий шаг в `path` до того, как `MovementSystem` обнаружит
пустой путь. Это и устраняет паузу.

### planMove
Нужна функция планирования **одного следующего шага** к цели (look-ahead):
A* до цели, взять `path[0]` как следующий шаг. Существующий `planMoveToPoint`
строит полный путь — для look-ahead берём из него первый шаг либо добавляем
тонкую обёртку `planNextStep(droneId, target, ...)`.

## Поток данных (типичный шаг, цель не меняется)

```
тик N:  MovementSystem шаг в клетку C → path стал пуст → drone:moved
        ↓ (~15ms ранний resume)
        Driver: resume → воркер
        ↓
        воркер: intent moveTo(same target)
        ↓
        Driver: та же цель → planNextStep → path = [следующая клетка]
тик N+1: MovementSystem: path не пуст → продолжает движение без паузы
```

Гонка: между «path опустел → state=running» и приходом intent. При обычной
latency (~15ms ≪ длительность шага: speed=1 → 1000ms, speed=10 → 100ms) intent
успевает, path продлевается до физической остановки. Если не успел (cold start)
— дрон стоит на клетке, resume придёт следующим тиком (безопасно, без рывка).

## Граничные случаи

- **Cold start / лаг воркера** → дрон останавливается на клетке, ждёт ответа.
  Пауза возможна только при реальном лаге (редко), не на каждой клетке.
- **Смена цели на лету** → приходит `moveTo(другая точка)` → driver строит новый
  путь от текущей клетки → дрон плавно меняет направление со следующего шага.
- **Действие ≠ move** → `state` выставляется (mine/charge/drop), дрон стоит на
  клетке. `onDroneMoved` для не-moveTo не шлёт ранний resume (уже так).
- **Блокировка (drone:blocked)** → путь очищен, `state=running`, ранний resume
  подавлён флагом `blocked` (уже так).
- **Достижение цели** → следующий `moveTo` вернёт пустой путь (уже на месте) /
  программа перейдёт к другому действию.

## Тестирование (TDD)

- **MovementSystem** (unit): шаг при непустом продлённом path → продолжает
  (`state=move`); шаг при пустом path → `state=running`, стоит на клетке.
- **CodeBehaviorDriver** (unit): `intent moveTo(same target)` в движении →
  дописывает шаг, не сбрасывает progress; `moveTo(другая цель)` → новый путь;
  не-move действие → не дописывает; cold start (intent не пришёл) → дрон стоит.
- **planNextStep** (unit): возвращает следующий 1 шаг A* к цели.
- **E2E / визуальная верификация**: запустить игру, дрон с `while(true)
  moveTo(mine)` едет плавно без пауз и без рывков назад; смена цели на лету
  даёт плавный разворот.

## Вне scope (YAGNI)

- Сущность `Path` (траектория из точек, итерация в цикле) — отдельная будущая
  фича.
- Подсветка рассчитанной траектории в игре — отдельная фича (зависит от
  хранения полного пути; в look-ahead модели полного пути нет).
- Физический deceleration (замедление) — на сетке не нужен; «плавность» =
  отсутствие пауз и рывков.
