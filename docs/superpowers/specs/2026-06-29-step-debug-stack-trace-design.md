# Step-режим отладки + stack-trace + подсветка строк в модулях

## Context

Сейчас у дронов есть Code Mode: код исполняется в Web Worker, между `await self.*` действиями
worker «зависает» на промисе, главный поток обрабатывает intent и шлёт `resume`. Уже реализована
инфраструктура подсветки текущей строки и **`lineStack`** (стек строк вызовов) — она forward-compatible
со step-режимом, но используется только частично:

- Подсветка работает **только для entry-программы**: пока исполнение внутри импортированного модуля,
  `mapStackToEntryLine` отбрасывает модульные строки и подсвечивает строку вызова (или гасит подсветку).
  См. `src/game/code/linker/mapLine.ts` — поведение «v1: гасить в модулях» явно зафиксировано в комментарии.
- **Нет step-режима**: пользователь не может пошагово прогонять код дрона и наблюдать состояние.
- Инструментация (`instrument.ts`) ставит `__line(N)` только перед `await self.*`, не на каждый statement.

Пользователь хочет: **step-режим отладки** + **stack-trace** + **подсветку строк внутри модулей**.

### Проблема, которую решаем
Отладка многомодульных программ дронов сейчас «слепая»: внутри модулей подсветка гаснет, стек
вызовов не виден, нельзя приостановить и прогнать код по шагам. Это первая фича, требующая
полноценного debugger-UX.

### Согласованные с пользователем решения
1. **Гранулярность шага — гибрид**: сначала step = одно действие дрона (`await self.*`), переиспользуя
   существующий intent-протокол; statement-level step (пауза на каждой строке) — **отдельный поздний этап**.
2. **UI — полноценная debug-панель**: Pause / Continue / Step + панель Call Stack (кадры `programId:line`,
   клик переходит к кадру в редакторе) + подсветка активной строки в каждом модуле.
3. **Область паузы — вся симуляция** (весь игровой цикл), не отдельный дрон. Переиспользуем существующие
   `GameController.pause()` / `GameController.step()` и `gameStatus === "paused"`.
4. **Breakpoints — пока нет** (только Pause/Step/Continue); вынесены в поздний этап (требуют statement-level).

### Желаемый результат
Выбрав дрона и встав на паузу, игрок видит: подсвеченную строку (в т.ч. внутри файла модуля, с
переключением активного редактора на нужный модуль), панель Call Stack с кадрами, и может прогонять
код дрона по одному действию за раз кнопкой Step.

---

## Декомпозиция на под-проекты

Задача большая → бьём на 3 независимо-поставляемых этапа. Этапы A и B самостоятельны и ценны сами по
себе; этап C — необязательное углубление.

| Этап | Что даёт | Зависит от |
|------|----------|-----------|
| **A. Stack-trace + подсветка в модулях** | Call Stack panel; подсветка активной строки внутри файлов модулей; переключение редактора на кадр | — (инфра `lineStack` уже есть) |
| **B. Step-режим по действиям дрона** | Pause/Continue/Step UI; пошаговый прогон кода дрона (шаг = одно `await self.*`) | A (для отображения позиции на паузе) |
| **C. Statement-level step + breakpoints** | Пауза на любой строке; точки останова | A, B |

**Эта спека детально описывает этапы A и B** (то, что просит пользователь). Этап C — намечен крупными
мазками для forward-compatibility, реализуется отдельной спекой позже.

## Этап A — Stack-trace и подсветка строк в модулях

### Цель
Заменить политику «гасить подсветку в модулях» на «показывать полный стек кадров и подсвечивать
активную строку в правильном файле».

### Модель данных
Сейчас `DroneState.currentLine: number | null` несёт одну строку entry-программы. Расширяем до стека кадров.

- Новый тип кадра (рядом с `mapLine.ts`):
  ```ts
  export interface StackFrame {
    programId: string;   // какой программе принадлежит кадр
    line: number;        // строка в ИСХОДНОМ коде этой программы (1-based)
  }
  ```
- Новая функция `mapStackToFrames(...)` в `src/game/code/linker/mapLine.ts`: маппит весь `lineStack`
  (склеенные строки) в массив `StackFrame[]`, **не отбрасывая** модульные кадры. Базируется на
  существующем `mapLine`, но без проверки `seg.programId !== entryId` — возвращает кадр для любого сегмента.
  - Самый внешний элемент стека = entry-кадр; самый глубокий = текущая активная строка.
- `mapStackToEntryLine` оставляем (для обратной совместимости / fallback), либо выражаем через
  `mapStackToFrames` (последний кадр) — решить при реализации, не плодя дублирование.

### Поток данных (расширение существующего)
`worker lineStack` → `WorkerMessage` (уже несёт `lineStack`) → `CodeBehaviorDriver` маппит в `StackFrame[]`
→ `Program.callStack` → `snapshotDrones` → `DroneState.callStack` → UI.

- `src/game/code/CodeBehaviorDriver.ts:189,195` — заменить `program.currentLine = mapToEntryLine(...)`
  на заполнение `program.callStack = mapStackToFrames(...)`. Сбрасывать в `finished/error/timeout`
  (строки ~202, 211, 251).
- `src/game/simulation/components/Program.ts` — добавить `callStack?: StackFrame[] | null`.
  `currentLine` оставить как производное (последний кадр) ради минимальных правок рендера — либо
  убрать и перевести всех потребителей на `callStack`. **Решение:** оставить `currentLine` как
  derived-геттер/поле (= последний кадр), чтобы не трогать лишний UI; `callStack` — источник правды.
- `src/shared/store/gameStore.ts` — `DroneState` получает `callStack: StackFrame[] | null`;
  `snapshotDrones` (~строка 36, и место заполнения) копирует его из `Program`.

### UI: вкладка DRONE как debugger — редактор следует за активным кадром
Основной режим (как step-into в браузерном дебаггере): на вкладке DRONE редактор **автоматически
показывает код той программы, в которой сейчас исполнение** (самый глубокий кадр стека), и подсвечивает
в ней активную строку. При заходе в подпрограмму редактор сам переключается на её код; при выходе —
возвращается к вызывающей.

- **Хлебные крошки стека вызова** над редактором:
  - Показывают весь `callStack` слева направо: внешний → текущий. Пример:
    `personal drone-3 → moveToBase → ...` (имя источника + цепочка программ/функций кадров).
  - Внешний элемент = откуда стартовали (personal/assigned программа дрона); далее — кадры подпрограмм.
  - Клик по крошке = выбрать кадр вручную (см. ниже) — переключает редактор на программу этого кадра.
  - Имя программы кадра — через `registry.get(frame.programId).name`.
  - Это самостоятельный компонент `src/ui/editor/CallStackBreadcrumbs/index.tsx` (или часть `CallStackPanel`).
  - Селекторы — `data-testid`, напр. `data-testid="callstack-crumb-{i}"`.

- **Какой код показывает редактор DRONE:**
  - По умолчанию следует за активным (самым глубоким) кадром — «follow execution».
  - Если пользователь кликнул по крошке/кадру — фиксируется на выбранном кадре, пока активная строка не
    уйдёт за пределы стека (тогда снова follow). Хранить «выбранный кадр» в локальном состоянии вкладки.
  - Когда активный кадр принадлежит модулю (не personal/assigned дрона) — редактор открывает код этого
    модуля (read-only либо редактируемо — решить при дизайне; код модуля редактируется на вкладке PROGRAM,
    поэтому в DRONE при step-into логично показывать его read-only превью с подсветкой).

- **Подсветка строки:**
  - `src/ui/editor/CodeEditor/CodeEditor.tsx` уже умеет `highlightLine`. Для отображаемого кадра берём
    из `callStack` кадр с нужным `programId` и подсвечиваем его `line`.
  - Текущее место в `ProgramEditor/index.tsx`, где `highlightLine` берётся только для активной
    программы дрона (assigned/personal), заменяется логикой «строка отображаемого кадра».

- **Открытый вопрос (дизайн):** разместить хлебные крошки как горизонтальную полоску над редактором
  vs отдельная вертикальная панель Call Stack сбоку. **Лучше выбрать через дизайн-сессию в браузере**
  (визуальное сравнение макетов) — см. отметку в этапе B.

### Тесты этапа A
- Unit (`mapLine.test.ts` рядом): `mapStackToFrames` для стека `[entryLine, moduleLine]` возвращает 2 кадра
  с правильными `programId`/`origLine`; для чисто-entry стека — 1 кадр; пустой/нулевой стек — `[]`.
- Unit `CodeBehaviorDriver` (если есть существующий тест): `program.callStack` заполняется при intent/wait,
  сбрасывается при finished/error.
- E2E: запустить миссию с программой, импортирующей модуль; во время исполнения внутри модуля —
  хлебные крошки показывают ≥2 кадров, редактор DRONE автоматически показывает код подпрограммы и
  подсвечивает её строку; клик по внешней крошке возвращает к коду вызывающей программы. Селекторы —
  `data-testid`.

## Этап B — Step-режим по действиям дрона

### Цель
Дать игроку Pause / Continue / Step над симуляцией, где **один Step = один игровой тик** до следующего
обработанного действия дрона, с остановкой и показом позиции (кадров из этапа A).

### Ключевое наблюдение
`GameController` **уже умеет** `pause()` (останавливает `GameLoop`) и `step()` (один `tick()` + проверка
условий), а `gameStatus` уже включает `"paused"`. Значит на уровне симуляции механика паузы есть —
этап B это в основном **UI debug-панели** поверх существующего, плюс уточнение семантики «шага».

### Семантика шага
- **Step (по действию)**: проматывать тики до тех пор, пока активный (выбранный) дрон не обработает
  следующее действие — т.е. пока его `callStack`/`currentLine` не изменится на следующее `await self.*`.
  - Простейшая реализация: Step вызывает `controller.step()` (один тик) повторно, пока у выбранного дрона
    `program.state` не вернётся к `running` и не сменится строка, либо до разумного лимита тиков
    (защита от зависания). Альтернатива — одиночный `controller.step()` (= один тик) как «micro-step».
  - **Решение:** дать обе кнопки — **Step (tick)** = один игровой тик (уже есть `controller.step()`), и
    **Step over (action)** = прогон тиков до следующего действия выбранного дрона. Action-step реализуется
    в `GameController` новым методом `stepDroneAction(droneId)` с лимитом тиков.

### UI: Debug controls
- Кнопки Pause/Continue (переиспользуют `controller.pause()`/`start()`), Step tick, Step action.
  Связать с `GameController` через тот же путь, что текущие Play/Pause (см. где монтируется
  `SimControls` и как пробрасывается `controller`).
- На паузе хлебные крошки стека (этап A) и подсветка показывают актуальную позицию выбранного дрона.

> **⚠ Дизайн-сессия в браузере (обязательно перед реализацией UI).** Перед версткой debug-controls
> провести визуальную дизайн-сессию (companion в браузере), чтобы сравнить макеты и выбрать:
> - расширять `src/ui/controls/SimControls.tsx` vs отдельная debug-секция/панель;
> - видимость debug-кнопок: всегда vs только при `gameStatus === "paused"`;
> - размещение хлебных крошек стека (полоса над редактором) vs вертикальная Call Stack панель сбоку
>   (этот же выбор из этапа A).
> Решение фиксируется в спеке перед написанием кода UI.

### GameController
- `src/game/GameController.ts`: добавить `stepDroneAction(droneId: EntityId): void` —
  цикл `tick()` пока действие выбранного дрона не сменится, с `MAX_STEP_TICKS` лимитом; затем
  `checkConditions()`. Переиспользует приватный путь `onTick`-логики (вынести общее, не дублировать).

### Тесты этапа B
- Unit `GameController`: `stepDroneAction` продвигает симуляцию до смены действия дрона и не превышает лимит
  тиков; на `won/failed` — no-op (как существующий `step()`).
- E2E: пауза → Step action → позиция дрона/подсветка сдвигается на следующее действие; Continue возобновляет.

## Этап C — Statement-level step + breakpoints (НЕ в этой итерации, набросок)

Для forward-compatibility фиксируем, чтобы A/B не закрывали путь:
- Инструментация: расширить `instrument.ts` — вставлять `await __step(N)` перед каждым statement (комментарий
  об этом уже есть в файле, строки 9–12). `__step` в `codeRuntime.ts` шлёт `intent`-подобное сообщение и
  зависает до `resume` — паузу на любой строке даёт уже существующий resume-механизм воркера.
- Breakpoints: набор `(programId, line)` → перед `__step` проверять, не брейк ли это; синхронизировать
  множество брейков в worker через `DriverMessage`.
- Step into/over/out: используют глубину `lineStack` (push/pop уже есть в `__call`).
- **A/B ничего из этого не ломают**: `callStack`/кадры и debug-панель переиспользуются 1:1; меняется только
  гранулярность остановок и добавляется протокол брейкпоинтов.
