# Единый процесс ведения документации фич — дизайн

**Дата:** 2026-06-29
**Тип:** процесс / документация (без кода фич)

## Проблема

Документация фич ведётся в четырёх параллельных потоках, которые пересекаются и
дублируют друг друга, а иногда не покрывают фичу вовсе:

- `docs/superpowers/specs/` — дизайн-спеки (генерит brainstorming-скилл), датированы.
- `docs/superpowers/plans/` — планы реализации (генерит writing-plans), датированы.
- `docs/features/` — `planned/` + `done/` + `index.md`, бессрочная карточка фичи
  (Зачем/Поведение/Критерии/Тех.заметки), ведётся вручную.
- `docs/sessions/` — журнал «цель + результаты» по каждой сессии (~80 файлов).

Симптомы:

1. **Дубли описания.** Карточка фичи дословно повторяет «Зачем/Поведение/Тех.заметки»
   из спеки; сессия пересказывает то же ещё раз.
2. **Пропуски.** Фича «подсветка вызова подпрограммы / step-debug» существует только как
   спека `2026-06-29-subprogram-call-highlight-design.md` (+ план + сессия) и **не заведена**
   в `docs/features/`. Источника правды для неё нет.
3. **Битые/внешние ссылки.** `features/done/library-modules-import.md` ссылается на план
   `.claude/plans/hidden-kindling-gizmo.md` — вне репозитория, недоступен.
4. **Шум.** Большинство сессионных файлов дублируют спеку/план/карточку и не находятся
   агентом при старте новой сессии (их 80, релевантность по имени не видна).

Корень: spec и plan создаются скиллами автоматически как часть workflow superpowers — это
не отключить и не нужно. А `features/` ведётся вручную параллельно — отсюда и дубли, и
пропуски. Схема должна принять, что specs/plans будут создаваться скиллами, и чётко
определить роль `features/` и `sessions/` относительно них.

## Решение: роли каталогов

| Каталог | Роль | Срок жизни | Кто ведёт |
|---|---|---|---|
| `docs/features/` | **Источник правды.** Одна фича = один файл-карточка: актуальное «что/зачем/поведение/критерии», статус, ссылки на spec/plan/сессии. `index.md` — единственный реестр. | Бессрочный, поддерживается актуальным | Вручную (агент по правилам CLAUDE.md) |
| `docs/superpowers/specs/` | Дизайн-артефакт сессии (brainstorming). Датированный снимок «как придумали тогда». | Исторический, **не правится задним числом** | Авто (brainstorming-скилл) |
| `docs/superpowers/plans/` | План реализации (writing-plans). Датированный. | Исторический | Авто (writing-plans-скилл) |
| `docs/sessions/` | **Опциональный журнал решений и тупиков.** Пишется только при наличии нарратива сверх карточки. Ссылается, не пересказывает. | Исторический | Вручную, по необходимости |

## Принцип «один факт — одно место»

- **Описание фичи** (Зачем/Поведение/Критерии) — только в карточке. Spec и session
  **ссылаются**, не копируют.
- **«Как придумали»** — в спеке. Карточка даёт ссылку, не пересказывает дизайн целиком.
- **«Как делали по шагам»** — в плане.
- **«Что пошло не так / что отброшено»** — в сессии (этого нет ни в git, ни в спеке).
- Карточка — навигационный хаб: в шапке ссылки на spec/plan, внизу секция «Журнал сессий».

## Главное правило процесса (закрывает дыру step-debug)

> **Каждая фича обязана иметь карточку в `docs/features/`.**
> brainstorming/writing-plans создают spec и plan как обычно, но фича заводится карточкой,
> которая линкует на них (при старте реализации либо по завершении). Spec без карточки —
> незавершённый процесс.

## Роль и правила `sessions/`

Журнал смещается с «отчёт о каждой сессии» на **«журнал решений и тупиков»**:

- Писать сессию, **только когда есть нетривиальный нарратив**: отброшенные подходы,
  найденный root-cause бага, неочевидное «почему так, а не иначе». Сессия вида
  «реализовал по плану, всё зелёное» **не пишется** — это уже в карточке/плане/git.
- Сессия **ссылается** на карточку/спеку, не пересказывает их.
- Находимость: карточка фичи держит секцию «Журнал сессий» со ссылками на все сессии по
  этой фиче (линковка цепочки сессий — через карточку, не prev/next между сессиями).
- Статистика (сколько сессий, по каким фичам) сохраняется самим списком файлов.

## Шаблон карточки фичи (новый)

```markdown
# <Название фичи>

**Статус:** planned | in-progress | done
**Spec:** docs/superpowers/specs/<date>-<topic>-design.md   (если есть)
**Plan:** docs/superpowers/plans/<date>-<topic>.md           (если есть)

## Зачем
## Поведение
## Критерии готовности
## Технические заметки   (кратко; детали — в спеке/плане по ссылке)

## Журнал сессий
- docs/sessions/<date>-... — <что произошло>
```

Поля Spec/Plan/«Журнал сессий» опускаются, если артефактов нет (мелкие фичи без brainstorming).

## Изменения в CLAUDE.md

**Раздел «Feature planning»:**

- Карточка `docs/features/<slug>.md` — единственный источник правды описания фичи.
- Любая фича обязана иметь карточку; карточка линкует на spec/plan (если есть).
- specs/ и plans/ — исторические артефакты сессий, не правятся задним числом, не реестр.
- `planned → done` = перемещение файла + строка в `index.md` (как сейчас).
- Один факт — одно место: не дублировать описание из карточки в spec/session, и наоборот.

**Раздел «Documentation»:**

- Сессия — **опциональна и нарративна**, не «в конце каждой сессии». Пишется только при
  наличии решений/тупиков/нетривиального нарратива.
- Сессия запрещает дублировать карточку/спеку — только ссылка + то, чего нет нигде.
- Карточка фичи собирает ссылки на сессии в секции «Журнал сессий».

## Миграция (объём: «новые + проблемные»)

1. `features/done/library-modules-import.md` — заменить битую ссылку на внешний план
   реальной спекой; добавить шапку Spec/Plan и «Журнал сессий».
2. Завести недостающую карточку для step-debug / subprogram-call-highlight в `features/done/`,
   залинковать на существующие spec + plan + session. Добавить в `index.md`.
3. Обновить `docs/features/FEATURE-TEMPLATE.md` и `docs/features/index.md` под новый шаблон.
4. **Не трогаем:** старые карточки `done/` (кроме п.1), весь `specs/`, весь `plans/`,
   старые `sessions/`. Новый шаблон применяется вперёд. Реестры `specs/index.md`/`plans/index.md`
   **не заводим** — единственный реестр остаётся `features/index.md`.

## Не входит

- Массовая миграция всех ~15 карточек `done/` (только проблемные).
- Реестры/индексы для specs и plans.
- Цепочки prev/next между сессиями (линковка только через карточку).
- Автоматизация/хуки для проверки «у каждой фичи есть карточка» (правило в CLAUDE.md, не код).
