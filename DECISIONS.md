## Архитектура

**[26-06-2026] Движение дрона continuous: путь не зачищается после шага, дрон едет по всему буферу пути** — раньше `MovementSystem` после каждого шага делал `movement.path = []` (атомарный шаг — одна команда = одна клетка). Теперь путь сохраняется: дрон проходит весь `path` подряд, клетки снимаются через `path.shift()`. Это убирает паузу-рывок между клетками при ходьбе по длинному маршруту и резкий сброс прогресса. Сопутствующие изменения: `CodeBehaviorDriver` при новом `moveTo` во время движения вызывает `extendPathTail` (пересчитывает хвост пути от `path[0]`, не трогая текущий шаг и `progress`) вместо `planMoveToPoint` (полный пересброс) — см. `planMove.ts`. Для не-`moveTo` действий (mine/drop/charge) путь обрезается до текущего шага (`path.length = path.length ? 1 : 0`), чтобы дрон доехал до начатой клетки и не дёргался. Это отступление от решения [25-05-2026] «действия атомарны» в части движения: атомарность контроля сохраняется на уровне команд программы, но физический проход маршрута больше не разбивается на отдельные тики-команды.

**[26-06-2026] Разрешение коллизий движения через резервирование целевой клетки (поле `Movement.reserved`)** — когда два дрона едут в одну клетку, раньше оба копили `progress` (визуально «сливались»), затем один доезжал, а второго отбрасывало назад со сбросом `path`/`progress`. Теперь: дрон при ВЫЕЗДЕ (`progress === 0`) проверяет, свободна ли целевая клетка (`path[0]`); занятой считается клетка, где стоит дрон ИЛИ которую зарезервировал движущийся дрон (`progress > 0 && reserved`). Если занята — дрон не начинает движение вовсе (стоит, `progress` остаётся 0, путь сохраняется), вместо накопления прогресса впустую. Если свободна — резервирует её (`reserved = path[0]`) на всё время поездки, по прибытии снимает резерв. Так нет ни «слияния», ни «отбрасывания». Встречный обмен местами (swap) запрещён явной проверкой. Колонна вплотную трогается по очереди: клетка освобождается только по факту прибытия, поэтому за тик сдвигается лишь головной дрон, следующий — на следующем тике. Дедлок-циклы (A→B→A) корректно остаются стоять. Решение выбрано под профиль «мало дронов (2-6), просторно» — полноценный reservation table (Cooperative A* с временными слотами) избыточен. Визуальный аспект: в `GameScene` ждущий дрон (`reserved == null`, `progress == 0`) рисуется без t-интерполяции к целевой клетке (`targetCell = null`), иначе доля кадра `t` дёргала бы его вперёд-назад (тряска). Альтернатива «отбрасывать заблокированного назад + событие `drone:blocked`» отвергнута — давала именно тот визуальный артефакт, который чинили; событие `drone:blocked` оставлено в типах, но больше не эмитится.

**[20-06-2026] `optimizeDeps.exclude: ["monaco-editor"]` условна: только в dev (`npm run dev`), не в CI (`npm run build`)** — Monaco регистрирует воркеры через `MonacoEnvironment.getWorker`, импортируя их как `?worker` (Vite-специфичный суффикс). В dev-режиме Vite использует esbuild для пребандлинга: если не исключить Monaco, esbuild не поймёт Vite-суффиксы внутри node_modules, воркеры превратятся в обычные модули — Monaco сломается. `exclude` отключает пребандлинг в dev, и Monaco грузит 2300+ ESM-модулей самостоятельно (медленнее, но корректно). В CI (`npm run build`) production-сборка использует Rollup, который корректно обрабатывает `?worker` — `exclude` не нужен, и Rollup может оптимизировать Monaco. Реализация: `const isCI = !!process.env.CI` в vite.config.ts, `optimizeDeps` остаётся пустой в CI (` {}`) и содержит `exclude` локально. CI/e2e уже запущены на production-сборке через `vite preview` (см. playwright.config.ts, .github/workflows/deploy.yml).

**[10-06-2026] Источник поведения дрона абстрагирован в «behavior driver»; код и блоки сходятся к единой модели намерений** — вводится возможность программировать дронов настоящим JS/TS (async/await: `await drone.moveTo()`), не удаляя визуальные блоки. Тип поведения — дискриминированное объединение `DroneBehavior = { source:'block'; instructions } | { source:'code'; code }`, однозначное по `source`. `ProgramExecutionSystem` каждый тик выбирает driver: `AstBehaviorDriver` (обёртка над текущим `stepProgram`) или `CodeBehaviorDriver` (исполняет JS в Web Worker). Ключ: `stepProgram` уже работает как модель намерений — действие выставляет `program.state` и `return`, системы доводят его за тики. Код игрока встаёт на тот же контракт: `await drone.action()` шлёт намерение из воркера, driver выставляет `state` и выходит, по завершении (`state→idle`) резолвит промис. Так системы Movement/Mining/Energy, `state`-механизм и астар не меняются, а детерминизм пошаговости сохраняется. Воркер даёт изоляцию и таймаут против вечных циклов (`while(true){}` без await). Переключатель код/блоки — в настройках, только вне миссии (у дрона живое состояние исполнения). Аудитория — ученики-фронтендеры. Альтернативы отвергнуты: tick-функция (Screeps-стиль) — менее идиоматична для фронтендеров; внешняя IDE как runner — тяжёлый онбординг, нет доступа к ФС из браузера, ломает самодостаточность игры. Спецификация: `docs/superpowers/specs/2026-06-10-drone-code-mode-design.md`.

**[10-06-2026] Реализация ядра Code Mode: CodeWorkerPort, codeRuntime, CodeBehaviorDriver** — `BehaviorDriver` — общий интерфейс (`step(droneId, ctx)`); `AstBehaviorDriver` — обёртка над `stepProgram` без изменения поведения. `CodeBehaviorDriver` держит per-drone worker-сессию за абстракцией `CodeWorkerPort` (`BrowserWorkerPort` поверх `Worker` в проде, `NodeWorkerPort` поверх `node:worker_threads` + `tsx` в Vitest — оба запускают общий `codeRuntime`, который гоняет код игрока как `AsyncFunction`). Протокол: `intent`/`wait`/`finished`/`error` от воркера, `start`/`resume` от driver'а; `drone.wait(seconds)` обрабатывается полностью на стороне driver'а (не требует похода в воркер каждый тик). Таймаут синхронного участка между `await` — `setTimeout` в driver'е + `port.terminate()`. Сенсоры — снапшот `SensorsSnapshot` (`collectSensors`), формируется на старте каждого resume, обеспечивает консистентность геттеров `drone.*`/`distance()`/`deposit()` в пределах тика. Этап 1: код передаётся через временное поле `program.codeSource` (additive в `ProgramComponent`); `ProgramExecutionSystem` выбирает driver по наличию `codeSource`. Полная миграция `ProgramRegistry`/`ProgramDef` на дискриминированный `DroneBehavior` (`source: 'block'|'code'`) — отдельная будущая задача после Monaco (этап 2), т.к. затрагивает 14 файлов (миссии, UI редактор, рендерер, store).

**[15-06-2026] Миграция реестра программ на `ProgramDef.behavior: DroneBehavior`** — `ProgramDef` больше не содержит `instructions`/`behaviorMode`/`codeSource` напрямую: единственное поле `behavior: DroneBehavior`, дискриминированное объединение `{ sourceForm: 'block'; instructions: Instruction[] } | { sourceForm: 'code'; code: string }`. Тег назван `sourceForm` (не `source`, как в черновом варианте этапа 1) — по явному запросу пользователя, чтобы не путать с другими "source" в проекте. Затронуты: `types.ts`, `interpreter.ts`/`AstBehaviorDriver`/`CodeBehaviorDriver`/`ProgramExecutionSystem`, миссии (`mission1-4.ts`), `gameStore.ts` (фильтрация программ, мутации инструкций, переключение personal-программы код/блоки, `createProgram`), `ProgramEditor` (UI редактор блоков и кода), все тестовые литералы `ProgramDef` по проекту. `program.codeError` остался в `ProgramComponent` как runtime-состояние исполнения, не часть статического определения программы — миграция его не затрагивает. Все обращения к старым полям убраны (`git grep` чист), `npm test`/`npm run type-check`/`npm run build` зелёные.

**[31-05-2026] Состояние дрона кодируется одним полем `state`, без `waitingFor`** — `ProgramState` расширен до `'idle' | 'running' | 'move' | 'mine' | 'drop' | 'charge'`. Раньше «занят действием» выражалось двумя полями (`state = 'waiting'` + `waitingFor = 'move' | 'mine' | 'drop' | 'charge'`) — скрытый инвариант, не гарантируемый типами, и избыточность (один логический факт в двух полях). Теперь проверка прямая (`state === 'move'` вместо `state === 'waiting' && waitingFor === 'move'`), сброс — одна строка (`state = 'running'`), а UI-метки генерируются без маппинга (`state.toUpperCase()`). Имена action-состояний намеренно совпадают с именами инструкций языка (`MOVE_TO`→`move`, `MINE`→`mine`, `DROP`→`drop`, `CHARGE`→`charge`) — это убирает лишний слой соответствия команда↔состояние. Альтернатива «оставить два поля» отвергнута: дублирование без выгоды.

**[26-05-2026] Условие IF строится из вызовов функций реестра, не из захардкоженных kind** — leaf теперь `FunctionCall <op> Operand`, где `Operand` — число или вложенный `FunctionCall` глубиной 1. Шесть MVP-функций (`Energy`, `EnergyMax`, `Inventory`, `InventoryMax`, `Deposit`, `Distance`) живут в `src/game/programs/functions.ts`; новая функция добавляется одной записью в реестр, UI подхватывает автоматически. Альтернатива «расширять `ConditionProperty.kind` под каждый кейс» отвергнута: она требовала бы хардкодить пары «свойство × сравниваемое поле» (например, отдельный `INVENTORY_NOT_FULL`) и не давала бы `Distance = 0` или `Inventory < InventoryMax`. Аргумент функции — `Self` или конкретная сущность миссии; `null` от `evaluate` → leaf `false`.

**[25-05-2026] Действия дронов атомарны — одна команда = одна единица работы** — команды `MOVE_TO`, `MINE`, `DROP`, `CHARGE` выполняют ровно один тик действия (одна клетка / одна руда / одна единица энергии) и сразу возвращают программу в `running`. Это даёт игроку точку контроля между шагами: между «добыл первую руду» и «добыл вторую» можно вставить `IF` или другую инструкцию. Макроповедение строится через `LOOP` / `REPEAT`. Альтернатива «команда доводит работу до конца» отвергнута: она упрощает первые миссии, но закрывает дверь к интересным условным программам и делает существующие `LOOP`/`IF` декоративными. Пассивная зарядка на станции (без активной `CHARGE`) оставлена как свойство станции, а не команды.

**[25-05-2025] Simulation Layer не импортирует Phaser** — чистая игровая логика тестируется без движка. Phaser используется только как рендерер, наблюдающий за состоянием.

**[15-05-2025] Zustand как UI-стор** — хранит только то, что нужно React для отрисовки UI. Игровое состояние живёт в симуляции, Zustand получает снапшоты.

**[12-05-2025] Поток данных однонаправленный** — `React UI → команда → Simulation → Phaser рендерит → Zustand обновляет UI`. Phaser не изменяет состояние симуляции напрямую.
