## Архитектура

**[10-06-2026] Источник поведения дрона абстрагирован в «behavior driver»; код и блоки сходятся к единой модели намерений** — вводится возможность программировать дронов настоящим JS/TS (async/await: `await drone.moveTo()`), не удаляя визуальные блоки. Тип поведения — дискриминированное объединение `DroneBehavior = { source:'block'; instructions } | { source:'code'; code }`, однозначное по `source`. `ProgramExecutionSystem` каждый тик выбирает driver: `AstBehaviorDriver` (обёртка над текущим `stepProgram`) или `CodeBehaviorDriver` (исполняет JS в Web Worker). Ключ: `stepProgram` уже работает как модель намерений — действие выставляет `program.state` и `return`, системы доводят его за тики. Код игрока встаёт на тот же контракт: `await drone.action()` шлёт намерение из воркера, driver выставляет `state` и выходит, по завершении (`state→idle`) резолвит промис. Так системы Movement/Mining/Energy, `state`-механизм и астар не меняются, а детерминизм пошаговости сохраняется. Воркер даёт изоляцию и таймаут против вечных циклов (`while(true){}` без await). Переключатель код/блоки — в настройках, только вне миссии (у дрона живое состояние исполнения). Аудитория — ученики-фронтендеры. Альтернативы отвергнуты: tick-функция (Screeps-стиль) — менее идиоматична для фронтендеров; внешняя IDE как runner — тяжёлый онбординг, нет доступа к ФС из браузера, ломает самодостаточность игры. Спецификация: `docs/superpowers/specs/2026-06-10-drone-code-mode-design.md`.

**[31-05-2026] Состояние дрона кодируется одним полем `state`, без `waitingFor`** — `ProgramState` расширен до `'idle' | 'running' | 'move' | 'mine' | 'drop' | 'charge'`. Раньше «занят действием» выражалось двумя полями (`state = 'waiting'` + `waitingFor = 'move' | 'mine' | 'drop' | 'charge'`) — скрытый инвариант, не гарантируемый типами, и избыточность (один логический факт в двух полях). Теперь проверка прямая (`state === 'move'` вместо `state === 'waiting' && waitingFor === 'move'`), сброс — одна строка (`state = 'running'`), а UI-метки генерируются без маппинга (`state.toUpperCase()`). Имена action-состояний намеренно совпадают с именами инструкций языка (`MOVE_TO`→`move`, `MINE`→`mine`, `DROP`→`drop`, `CHARGE`→`charge`) — это убирает лишний слой соответствия команда↔состояние. Альтернатива «оставить два поля» отвергнута: дублирование без выгоды.

**[26-05-2026] Условие IF строится из вызовов функций реестра, не из захардкоженных kind** — leaf теперь `FunctionCall <op> Operand`, где `Operand` — число или вложенный `FunctionCall` глубиной 1. Шесть MVP-функций (`Energy`, `EnergyMax`, `Inventory`, `InventoryMax`, `Deposit`, `Distance`) живут в `src/game/programs/functions.ts`; новая функция добавляется одной записью в реестр, UI подхватывает автоматически. Альтернатива «расширять `ConditionProperty.kind` под каждый кейс» отвергнута: она требовала бы хардкодить пары «свойство × сравниваемое поле» (например, отдельный `INVENTORY_NOT_FULL`) и не давала бы `Distance = 0` или `Inventory < InventoryMax`. Аргумент функции — `Self` или конкретная сущность миссии; `null` от `evaluate` → leaf `false`.

**[25-05-2026] Действия дронов атомарны — одна команда = одна единица работы** — команды `MOVE_TO`, `MINE`, `DROP`, `CHARGE` выполняют ровно один тик действия (одна клетка / одна руда / одна единица энергии) и сразу возвращают программу в `running`. Это даёт игроку точку контроля между шагами: между «добыл первую руду» и «добыл вторую» можно вставить `IF` или другую инструкцию. Макроповедение строится через `LOOP` / `REPEAT`. Альтернатива «команда доводит работу до конца» отвергнута: она упрощает первые миссии, но закрывает дверь к интересным условным программам и делает существующие `LOOP`/`IF` декоративными. Пассивная зарядка на станции (без активной `CHARGE`) оставлена как свойство станции, а не команды.

**[25-05-2025] Simulation Layer не импортирует Phaser** — чистая игровая логика тестируется без движка. Phaser используется только как рендерер, наблюдающий за состоянием.

**[15-05-2025] Zustand как UI-стор** — хранит только то, что нужно React для отрисовки UI. Игровое состояние живёт в симуляции, Zustand получает снапшоты.

**[12-05-2025] Поток данных однонаправленный** — `React UI → команда → Simulation → Phaser рендерит → Zustand обновляет UI`. Phaser не изменяет состояние симуляции напрямую.
